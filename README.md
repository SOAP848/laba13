Система состоит из трёх автономных агентов (микросервисов), взаимодействующих через брокер сообщений NATS:

1. **Генератор тестов** (`test-generator`) – создаёт тестовые сценарии на основе исходного кода.
2. **Запускатель тестов** (`test-runner`) – выполняет тесты в заданном окружении и собирает результаты.
3. **Анализатор покрытия** (`coverage-analyzer`) – анализирует покрытие кода тестами и выдаёт рекомендации.

## Архитектура

- Каждый агент – отдельный Go-микросервис.
- Общение через NATS (JetStream) с использованием JSON-сообщений.
- Поддерживается автоматическая цепочка: генерация → запуск → анализ.
- Оркестратор для демонстрации сквозного сценария.

## Быстрый старт

### Требования

- Go 1.21+
- Docker и Docker Compose (для запуска в контейнерах)
- Сервер NATS (запускается автоматически через docker-compose)

### Локальный запуск (без Docker)

1. Установите зависимости:
   ```bash
   go mod download
   ```

2. Запустите сервер NATS (например, через Docker):
   ```bash
   docker run -d -p 4222:4222 -p 8222:8222 nats -js
   ```

3. В отдельных терминалах запустите агентов:
   ```bash
   go run cmd/test-generator/main.go
   go run cmd/test-runner/main.go
   go run cmd/coverage-analyzer/main.go
   ```

4. Запустите оркестратор для демонстрации:
   ```bash
   go run cmd/orchestrator/main.go
   ```

### Запуск через Docker Compose

```bash
docker-compose up --build
```

Система развернёт:
- Сервер NATS на портах 4222 (клиенты) и 8222 (мониторинг)
- Трёх агентов (test-generator, test-runner, coverage-analyzer)
- Оркестратор, который выполнит сквозной сценарий и завершится

Логи можно наблюдать в выводах контейнеров.

## Взаимодействие агентов

### Сообщения

Все сообщения определены в `pkg/models/messages.go`. Основные типы:

- `test.generate` → `test.generated`
- `test.run` → `test.completed`
- `coverage.analyze` → `coverage.analyzed`

Агенты также подписаны на смежные события для автоматизации (например, анализатор слушает завершение тестов).

## Расширенная система (дополнительные возможности)

Система была расширена дополнительными компонентами для повышения гибкости, наблюдаемости и интеллектуальных возможностей:

### Дополнительные агенты

1. **Stateful Agent** (`stateful-agent`) – сохраняет состояние обработки в Redis, обеспечивая устойчивость к перезапускам.
2. **Аукционер** (`auctioneer`) – реализует модель торгов, где агенты "торгуются" за задачи на основе стоимости и навыков.
3. **Скалер** (`scaler`) – автоматически масштабирует количество экземпляров агентов на основе нагрузки в очередях NATS, используя Docker API.
4. **Python LLM Agent** (`python-llm-agent`) – интегрирует большие языковые модели (Groq API) для интеллектуального анализа кода и генерации тестов.
5. **Веб-панель** (`web-panel`) – предоставляет веб-интерфейс для мониторинга агентов, очередей и задач, а также ручного запуска задач.
6. **Streamlit-панель** (`streamlit-panel`) – современный интерактивный дашборд на Python Streamlit с графиками и управлением задачами.

### Распределённая трассировка (OpenTelemetry + Jaeger)

Все агенты инструментированы с помощью OpenTelemetry. Трассировка передаётся в Jaeger через OTLP-коллектор, что позволяет визуализировать полный путь выполнения задачи через цепочку агентов.

- Jaeger UI доступен по адресу: http://localhost:16686
- OTLP коллектор принимает данные на порту 4317

### Настройка Python LLM Agent

Для работы Python LLM агента требуется API ключ сервиса Groq. Вы можете получить ключ на [платформе Groq](https://console.groq.com/).

1. Создайте файл `.env` в корне проекта (или скопируйте `.env.example`):
   ```bash
   GROQ_API_KEY=your_api_key_here
   ```
2. При запуске через Docker Compose переменная будет автоматически подставлена в контейнер.

Если ключ не указан, агент будет работать в ограниченном режиме (без вызовов LLM).

```bash
docker-compose up --build
```

Система развернёт все сервисы:
- **NATS** (порты 4222, 8222)

- **Веб-панель**: http://localhost:8000 – просмотр состояния агентов, очередей, задач, ручной запуск задач.
- **Streamlit-панель**: http://localhost:8501 – современный интерактивный дашборд с графиками, фильтрами и управлением задачами.

## Расширение системы

### Добавление нового агента

1. Создайте директорию в `cmd/` с `main.go`.
2. Используйте `natsclient.NewClient()` для подключения к NATS.
3. Подпишитесь на нужные топики и реализуйте обработку.
4. Добавьте сборку в Dockerfile и сервис в docker-compose.yml.

### Поддержка новых языков/фреймворков

- В агенте `test-generator` расширьте функцию `generateExampleTest()`.
- В агенте `test-runner` добавьте соответствующую команду запуска тестов.
- Обновите модели данных при необходимости.

## Мониторинг

- Агенты пишут логи в stdout (структурированные через logrus).

## Тестирование

Система включает интеграционные тесты, проверяющие работу всей цепочки агентов.

### Запуск интеграционных тестов

1. Убедитесь, что все сервисы запущены:
   ```bash
   docker-compose up -d
   ```

2. Выполните PowerShell-скрипт (Windows):
   ```powershell
   ./scripts/run_integration_tests.ps1
   ```

   Или bash-скрипт (Linux/macOS):
   ```bash
   ./scripts/run_integration_tests.sh
   ```

   Скрипт проверит:
   - Наличие всех запущенных контейнеров
   - Возможность отправки задачи через веб-панель
   - Обработку задачи агентами (увеличение счётчика в Redis)
   - Наличие трасс в Jaeger
   - Состояние потоков NATS JetStream

3. Запуск Go-интеграционных тестов (без Docker):
   ```bash
   go test ./tests/integration -v
   ```

   Эти тесты используют встроенный NATS-сервер и проверяют базовое взаимодействие сообщений.

### Структура тестов

- `tests/integration/full_pipeline_test.ps1` – PowerShell-скрипт для проверки полного пайплайна.
- `tests/integration/agent_integration_test.go` – Go-тесты для проверки взаимодействия через NATS.
- `scripts/run_integration_tests.ps1` – скрипт-обёртка для запуска всех тестов.
